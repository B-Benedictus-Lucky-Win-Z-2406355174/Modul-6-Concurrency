Milestone 1 Reflection notes
Pada milestone 1, saya baru pertama kali membuat server sederhana di Rust, jadi banyak hal yang benar-benar baru untuk saya. Awalnya saya belum paham kenapa harus memakai TcpListener, tapi setelah mencoba sendiri saya mengerti bahwa listener ini dipakai untuk menunggu koneksi dari browser di alamat 127.0.0.1:7878. Saya juga belajar bahwa setiap koneksi yang masuk dibaca sebagai stream, lalu diproses di fungsi handle_connection supaya kode utama lebih rapi dan mudah dibaca. Bagian yang paling menantang buat saya adalah membaca request HTTP dengan BufReader dan lines(), karena ternyata request dari browser itu dikirim sebagai teks per baris. Saya sempat bingung waktu lihat pesan compile dan warning, tetapi dari situ saya belajar bahwa pesan compiler justru membantu menunjukkan letak masalah lebih cepat. Setelah program dijalankan, saya bisa melihat isi request header di terminal, dan itu membuat saya lebih paham alur komunikasi dari browser ke server. Dari milestone ini saya merasa masih pemula, tetapi saya jadi lebih percaya diri karena sudah memahami dasar server dan cara membaca request secara manual.

Milestone 2 Reflection Notes
![Commit 2 screen capture](/assets/images/commit2.png)
Di milestone 2, saya belajar cara mengirim response HTTP yang benar ke browser, bukan cuma baca request. Saya jadi paham kalau response harus ada status line, header seperti Content-Length, lalu isi HTML. Awalnya saya sempat bingung kenapa halaman tidak tampil, ternyata path file dan format response harus tepat. Setelah berhasil, saya lebih mengerti alur dasar server mengirim halaman ke browser.

Milestone 3 Reflection Notes
![Commit 3 screen capture](/assets/images/commit3.png)
Di milestone 3, saya belajar membuat response server jadi lebih logis, yaitu membedakan halaman yang valid dan yang tidak valid. Sekarang server saya tidak selalu mengirim hello.html, tapi mengecek request dulu lalu memilih response 200 atau 404. Dari sini saya mulai paham bahwa request line seperti GET / HTTP/1.1 bisa dipakai untuk routing sederhana. Saya juga belajar refactoring kecil dengan memisahkan status_line dan filename supaya kodenya tidak berulang dan lebih mudah dibaca. Waktu mencoba route /bad, saya lihat server benar-benar mengembalikan 404, jadi saya lebih mengerti alur dasar validasi request. 

Milestone 4 Reflection Notes
Di milestone 4, saya menambahkan simulasi request lambat dengan endpoint /sleep yang menunggu sekitar 10 detik sebelum memberi respons. Dari percobaan ini saya jadi paham kenapa server single thread bisa membuat semua request lain ikut menunggu. Saat satu tab membuka /sleep, tab lain yang membuka halaman utama juga terasa lambat, jadi efek blocking terlihat jelas. Awalnya saya kira ini masalah jaringan atau browser, tetapi ternyata sumber utamanya ada di server yang hanya memproses satu request pada satu waktu. Milestone ini membantu saya memahami kenapa konsep concurrency penting untuk aplikasi web.

Milestone 5 Reflection Notes
Di milestone 5, saya mengubah server jadi multithread dengan ThreadPool supaya beberapa request bisa diproses bersamaan. Buat saya yang masih pemula, bagian ini awalnya cukup membingungkan karena ada konsep channel, worker, dan closure. Setelah dicoba, saya paham bahwa thread utama hanya menerima koneksi, lalu pekerjaan dikirim ke worker melalui pool. Hasilnya terasa lebih baik karena ketika saya akses /sleep, request lain seperti / tidak ikut menunggu lama seperti di milestone 4. Saya juga belajar bahwa struktur kode jadi lebih rapi karena logika concurrency dipisahkan ke lib.rs. Dari milestone ini saya jadi lebih mengerti kenapa web server nyata butuh concurrency supaya tetap responsif saat ada request yang lambat.

Commit Bonus Reflection notes
Di bonus ini saya menambahkan fungsi build sebagai alternatif dari new untuk membuat ThreadPool. Dari sini saya belajar bahwa build yang mengembalikan Result lebih aman karena error bisa ditangani, terutama saat ukuran thread pool tidak valid seperti 0. Berbeda dengan new yang lebih ringkas tetapi bisa panic, build membuat alur error handling lebih jelas untuk caller. Saya juga jadi paham bahwa satu fitur bisa punya dua gaya API: satu untuk kemudahan, satu untuk keamanan. Untuk tetap membandingkan keduanya, saya biarkan new tetap ada dan memanggil build di dalamnya. Dari latihan ini, saya belajar dasar pemikiran desain API di Rust, bukan cuma membuat program berjalan.
